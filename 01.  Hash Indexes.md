

## The Problem Indexes Solve

Without an index, finding a row means scanning **every** row in the table → **O(N)** time.  
Indexes exist to avoid that scan.

---

## How Hash Indexes Work

|Concept|Simple Explanation|
|---|---|
|**Hash function**|A "black box" — give it a key (e.g. a name), it always spits out the same number|
|**Hash index**|Uses that number to jump _directly_ to where the data lives in memory|
|**Speed**|Because it jumps straight there: **O(1)** reads and writes|
|**Row pointer**|The index stores the key → physical disk address of that row|

### Handling Collisions

When two keys produce the same number:

- **Chaining** — store a linked list at that slot
- **Probing** — scan forward to the next free slot

---

## The Three Big Limitations

### 1. Must Live in RAM

- Hash functions scatter data _randomly_ → terrible for mechanical disks (lots of slow random jumps)
- So the entire hash index **must sit in RAM**
- RAM is expensive and limited → your keys must fit in available memory

### 2. Not Durable (Solved by the WAL)

RAM is wiped on power loss. The fix:

> **Write Ahead Log (WAL)** — every write is first appended to a sequential log on disk.  
> Sequential writes are fast. If the system crashes, replay the WAL to rebuild the index.

```
Write comes in → Append to WAL (disk, sequential) → Update hash index (RAM)
Crash happens  → Replay WAL → Hash index restored ✓
```

### 3. No Range Queries

Hash indexes **cannot** answer: _"give me all names between A and B"_

- You'd have to guess infinite keys, or scan the whole array → back to **O(N)**
- Use a **Binary Search Tree (BST)** instead for range queries → **O(log N)**

---

## Quick Trade-off Comparison

||Hash Index|BST Index|
|---|---|---|
|Single lookup|**O(1)** ✅|O(log N)|
|Range query|❌ Useless|**O(log N)** ✅|
|Storage|RAM only|Can live on disk|
|Durability|Needs WAL|More naturally durable|

---

## When to Use a Hash Index ✅

Use it when **all three** conditions are true:

1. You need the **fastest possible** single-row reads/writes (O(1))
2. Your keys are **small enough to fit in RAM**
3. Your app **never needs range queries**

---

## One-Line Summary

> A hash index is a RAM-resident hashmap that trades range-query ability for O(1) speed, protected from crashes by a Write Ahead Log.
