

## The Core Idea

2PL is a traditional approach to achieving **full serializability** — making concurrent transactions appear as if they ran one at a time.

> Historically favoured because CPUs were too slow for actual serial execution. Now both approaches exist.

---

## How It Works: Two Lock Modes

|Lock Type|Who can hold it|Purpose|
|---|---|---|
|**Reader Lock (Shared)**|Multiple transactions simultaneously|Safe concurrent reads|
|**Writer Lock (Exclusive)**|Only one transaction at a time|Modify a row|

### The upgrade rule:

To write, a transaction must **upgrade** its reader lock to a writer lock — which requires waiting for **all other readers** to release first.

```
Transaction wants to write a row:
  → Holds reader lock
  → Requests upgrade to writer lock
  → Must wait: other transactions still hold reader locks on that row
  → Once all readers release → exclusive writer lock granted → write proceeds
```

This prevents read-modify-update race conditions: the value you read can't be changed by anyone else while you hold a reader lock on it.

---

## The Big Problem: Deadlocks

**What it is:** Two transactions block each other indefinitely — each holds a lock the other needs.

> **Example:** Jordan and Snoop Dogg both update each other's shopping cart rows simultaneously.
> 
> - Both grab reader locks on rows A and B
> - Jordan tries to upgrade lock on row A → blocked by Snoop's reader lock
> - Snoop tries to upgrade lock on row B → blocked by Jordan's reader lock
> - Neither can proceed. **Deadlock.**

**Resolution:** The database detects the deadlock, **aborts one transaction**, and re-runs it later.

**Cost:** Deadlock detection + transaction abortion + re-execution = serious performance hit. Makes 2PL slow under high concurrency.

**Also:** Acquiring locks consumes resources, and 2PL often grabs locks defensively (more than strictly needed), degrading performance further.

---

## Phantoms & Advanced Locking

Standard 2PL can't handle **phantom writes** — you can't lock a row that doesn't exist yet.

Three solutions, in order of speed:

### 1. Predicate Locks

Lock all rows matching a **condition** (e.g. all classes named 'flexibility' at '6:00') — even rows that don't exist yet are covered by the condition.

- ✅ Precise — locks exactly the right rows
- ❌ Slow — evaluating conditions without a matching index is expensive

### 2. Index Range Locking

Lock a **broader superset** of rows using an existing index (e.g. lock _all_ 'flexibility' classes, not just the 6:00 one).

```
Predicate:  class = 'flexibility' AND time = '6:00'   (precise, slow)
Index lock: class = 'flexibility'  (all times)         (fast, broader)
```

- ✅ Fast — O(log N) lookup via index
- ❌ Locks more rows than necessary → can block unrelated queries

---

## Trade-off Summary

|Locking Strategy|Protects Against|Speed|Risk|
|---|---|---|---|
|Standard 2PL|Read-modify-update races|Moderate|Deadlocks|
|Predicate Locks|+ Phantoms|Slow|High evaluation cost|
|Index Range Locks|+ Phantoms|Faster|Over-locking unrelated rows|

---

## How This Fits the Isolation Picture

|Approach|Mechanism|Key Cost|
|---|---|---|
|Read Committed|Locks + old values|Some race conditions remain|
|Snapshot Isolation|Versioned rows|Write skew & phantoms remain|
|Actual Serial Execution|Single-threaded|Throughput capped at one core|
|**2PL**|**Shared + exclusive locks**|**Deadlocks, resource-heavy**|

---

## One-Line Summary

> Two-phase locking achieves serializability by making writes wait for all readers to finish — effective but prone to deadlocks and resource-heavy, with predicate or index range locks needed to also handle phantom writes.
