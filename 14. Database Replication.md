
## The Core Idea

**Database replication** = maintaining multiple copies of your data across several database nodes.

Essential for any large-scale application — a single database is a single point of failure.

---

## Why Replicate?

|Reason|Problem Solved|
|---|---|
|**Fault tolerance**|One database crashes → total data loss. Replicas allow rerouting|
|**Increased throughput**|One node can't handle massive read/write load. Multiple nodes share it|
|**Reduced latency**|One US database is slow for Australian users. Geolocate replicas near users|

---

## Timing Models: Synchronous vs. Asynchronous

### Synchronous (Strong Consistency)

Write is only confirmed to the client once **every replica** has acknowledged the update.

```
Client writes → Primary updates → Forwards to all replicas
  → Wait for all replicas to confirm
  → Only then: confirm write to client ✓
```

|||
|---|---|
|✅|No stale reads — all replicas always up to date|
|❌|Slow — must wait for network round-trip to every global replica|
|Use when|Stale reads are strictly unacceptable|

---

### Asynchronous (Eventual Consistency)

Primary confirms the write immediately, then updates replicas in the background.

```
Client writes → Primary updates → Immediately confirms to client ✓
  → Replicas updated in background (later)
```

|||
|---|---|
|✅|Fast writes — no waiting for replicas|
|❌|Clients may read stale data from a lagging replica|
|Use when|Speed matters and brief stale reads are acceptable (most web apps)|

---

## Implementation Methods: How Data Gets Replicated

### 1. Copying SQL Statements ❌

Send raw SQL commands (`INSERT`, `UPDATE`) to replicas.

**Problem:** Non-deterministic functions like `time.now()` produce different values on each node → data inconsistencies across replicas.

---

### 2. Write-Ahead Log (WAL) Shipping ❌

Replicate exact byte-level disk addresses from the WAL.

**Problem:** Tightly couples replicas to the same database software. A MySQL WAL means nothing to a PostgreSQL replica — different systems use different memory layouts.

---

### 3. Logical (Replication) Logs ✅ Industry Standard

Send row-level logical instructions: `"At row ID 1, set name = 'Jordan'"`.

```
Primary writes → Generates logical log entry (row ID + new values)
  → Sent to replicas over network
  → Any SQL database understands row IDs → applied correctly
```

|||
|---|---|
|✅|Works across different database systems (MySQL → PostgreSQL)|
|✅|Human-readable, debuggable|
|❌|Slight write overhead — database must generate the separate log|

---

## Full Comparison

||SQL Statement Copying|WAL Shipping|Logical Log|
|---|---|---|---|
|Cross-system compatible|❌|❌|✅|
|Deterministic|❌ (time.now() breaks it)|✅|✅|
|Write overhead|Low|Low|Slight|
|Industry standard|No|No|**Yes**|

---

## One-Line Summary

> Database replication duplicates data across nodes for fault tolerance, throughput, and latency — using asynchronous replication for speed in most apps, and logical replication logs as the standard method for cross-system compatibility.
