

**What it is:** A long-running read sees an inconsistent state because concurrent writes happen _during_ the read.

> **Example:** A query is summing all bank balances — the invariant is the total must always equal $1,000,000.
> 
> - Query reads Caitlyn's balance: **$100,000** ✓
> - Meanwhile: a transaction moves $100,000 from Chris → Caitlyn
> - Query then reads Chris's balance: **$0** (post-transfer)
> - Final sum: **$900,000** ← wrong. Money appears to have vanished.

The query wasn't wrong — it just read different rows at different points in time. A write interjected mid-read.

> Read skew is also called a **non-repeatable read** — if you re-read a value during the same query, you might get a different result.

---

## The Solution: Snapshot Isolation

Built on top of the **WAL** (which already sequences all writes for atomicity and durability).

### How it works

**Step 1 — Version every write**

- Every transaction hitting the WAL gets a **monotonically increasing sequence number** (1, 2, 3…)
- When a row is updated, the old value is **kept** alongside its transaction number — rows are never overwritten or deleted outright

**Step 2 — Read from a snapshot**

- When a query starts, it picks a **snapshot transaction number** (e.g. txn 15)
- For every row, the query gets the most recent value written **at or before txn 15**
- Any writes from txn 16, 17, 22… are completely **ignored**
- If a row was _created_ after txn 15, the query treats it as if it doesn't exist

```
Query starts at snapshot txn 15

Row A → last written at txn 12  →  visible ✅
Row B → last written at txn 15  →  visible ✅
Row C → last written at txn 17  →  invisible ❌ (after snapshot)
Row D → created at txn 20       →  invisible ❌ (didn't exist yet)
```

### Result

The query sees a **frozen, consistent view** of the database as it existed at one moment in time — concurrent writes can't interfere.

---

## Storage Cost

Keeping old row versions sounds expensive, but it's manageable:

- New versions are only stored when a row is **actually modified**
- Unchanged rows don't accumulate versions
- Old versions can be garbage collected once no active snapshot needs them

---

## How This Fits the Isolation Levels

|Isolation Level|Prevents|
|---|---|
|Read Committed|Dirty reads, dirty writes|
|**Snapshot Isolation**|**+ Read skew (non-repeatable reads)**|
|Serializable|+ All race conditions (most expensive)|

Each level builds on the previous — snapshot isolation is one step stronger than read committed.

---

## One-Line Summary

> Snapshot isolation fixes read skew by assigning every write a sequence number and keeping old row versions, so long-running reads can query a consistent historical snapshot without being affected by concurrent writes.
