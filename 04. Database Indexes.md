 
## The Core Trade-off

||Without Index|With Index|
|---|---|---|
|**Read**|O(N) — scan every row|O(log N) — binary search|
|**Write**|Fast — just append|Slower — append + update index|

An index is a **sorted structure** on a key. Sorting is what enables binary search and makes range queries efficient. The cost is that every write must also maintain that structure.

---

## Index Storage Types

### 1. Clustered Index

The **full row data** is stored directly inside the index.

- ✅ Fastest reads — find the key, you already have everything
- ❌ Expensive if you need multiple indexes — entire row duplicated into each one

---

### 2. Address Index (Non-Clustered)

Stores the **key + a pointer** (disk address) to where the row actually lives.

- ✅ Much less storage — practical to have many indexes on one table
- ❌ Slightly slower reads — must follow the pointer to get the full row

---

### 3. Covering Index

A **middle ground** — stores the key + a small subset of frequently accessed fields. Remaining fields are referenced by pointer.

```
Clustered     →  key + ALL fields
Covering      →  key + SOME fields + pointer
Address       →  key + pointer
```

Balances read speed against storage cost. Best when you always query the same few fields alongside the key.

---

## Multi-Dimensional (Composite) Indexes

Index on **more than one field**, with a hierarchical sort order.

### Example: Username → Date

```
Sort by Username first
  → Then, within each Username, sort by Date
```

This is fundamentally different from two separate indexes on Username and Date — a composite index creates **one unified sort order** that the database can traverse efficiently for queries using both fields together.

**Use when:** your application regularly queries or filters on multiple columns at once.

---

## Choosing the Right Index Type

||Clustered|Address|Covering|
|---|---|---|---|
|Read speed|**Fastest**|Slower (pointer hop)|Fast for queried fields|
|Storage cost|High (full row × N indexes)|Low|Medium|
|Multiple indexes?|❌ Impractical|✅ Easy|✅ Reasonable|
|Best for|One primary index|Many indexes|Frequent partial-row reads|

---

## The Full Picture — All Index Types

||Hash|B-Tree|LSM Tree|
|---|---|---|---|
|Lookup|**O(1)**|O(log N)|O(log N)|
|Range query|❌|✅|✅ (slower)|
|Write speed|Fast|Slower|**Fastest**|
|Storage|RAM|Disk|RAM + Disk|
|Best for|Fast single lookups|General purpose|Write-heavy workloads|

---

## One-Line Summary

> Indexes trade write speed for read speed — the key design decision is choosing _how_ the index stores data (clustered, address, or covering) and _what_ it indexes (single field or composite) based on your application's query patterns.
