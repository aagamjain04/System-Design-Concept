

## Row vs. Column Storage

||Row-Oriented|Column-Oriented|
|---|---|---|
|**How data is stored**|All fields of one row together|All values of one column together|
|**Best for**|Fetching full records (web apps)|Aggregations on specific fields (analytics)|
|**Example use case**|Loading a user's full profile|Finding min/max/avg across millions of rows|

**Why column storage is faster for analytics:** The disk arm moves less — all the values you need are stored continuously rather than scattered across rows.

---

## Key Advantages

### 1. Column Compression

Columns tend to have repeated or similar values → much easier to compress than rows.

**Bitmap Encoding** — represent each value as a series of 1s and 0s:

```
Value "London":  1 0 0 1 0 1 ...  (1 = present in this row, 0 = absent)
Value "Paris":   0 1 1 0 0 0 ...
```

**Run-Length Encoding** — compress the bitmap by replacing runs of repeated bits with a count:

```
Bitmap:  0 0 0 1 1  →  "3 zeros, 2 ones"  →  stored as: 32
```

**Benefits:** Less data over the network, more fits in RAM and CPU cache, faster queries.

---

### 2. Dictionary Compression

For columns with a small, finite set of string values.

```
5 distinct values → only need 3 bits to represent all of them (2³ = 8 combinations)
"London" → 001,  "Paris" → 010,  "Berlin" → 011 ...
```

Drastically reduces storage and improves cache performance.

---

### 3. Predicate Pushdown (Parquet)

**Parquet** is an open-source Apache format for columnar storage.

- Divides columns into **chunks**
- Stores lightweight **metadata** per chunk (min, max, sum, avg)

```
Query: find all values > 60

Chunk A: max = 55  →  skip entirely ✓  (no values can match)
Chunk B: max = 80  →  search this chunk
Chunk C: max = 45  →  skip entirely ✓
```

Skipping irrelevant chunks = significantly faster reads. This is called **predicate pushdown**.

---

## Downsides & Workarounds

### 1. Strict Sort Order Required

All column files must share the same sort order so values stay aligned across columns.

- **Workaround:** Duplicate the files with a different sort order for different query patterns — costs extra storage but speeds up specific queries.

---

### 2. Expensive Writes

Writing a single new row means dispersing values across many column files. Inserting mid-sort requires shifting existing data.

- **Workaround: LSM Tree buffer**

```
Incoming row write
  → Goes to in-memory LSM tree (fast)
  → When tree fills up → flush to disk in bulk
  → All column files written at once, sort order preserved
```

Batching writes through an LSM tree avoids the per-row cost entirely.

---

## Summary of Trade-offs

||Row-Oriented|Column-Oriented|
|---|---|---|
|Read (full record)|✅ Fast|Slow|
|Read (aggregate one field)|Slow|✅ Fast|
|Write (single row)|✅ Easy|Expensive (workaround: LSM buffer)|
|Compression|Weaker|✅ Much better|
|Best fit|Web apps, OLTP|Analytics, OLAP|

---

## One-Line Summary

> Column-oriented storage groups values by column rather than row, enabling faster analytics, better compression, and predicate-based chunk skipping — at the cost of expensive single-row writes, mitigated by buffering through an LSM tree.
