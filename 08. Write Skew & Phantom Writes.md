

## Problem 1: Write Skew

**What it is:** Two transactions each read a shared state, make a decision based on it, then write to _separate_ rows — breaking a system invariant in the process.

> **Example:** An ER database requires at least one doctor to be active at all times.
> 
> - Dr. Oz checks: 2 active doctors → safe to go on break → grabs lock on _his_ row → sets inactive
> - Dr. Toboggan checks: 2 active doctors → safe to go on break → grabs lock on _her_ row → sets inactive
> - Both writes succeed — they were on different rows, so neither blocked the other
> - Result: **0 active doctors.** Invariant broken.

The problem: row-level locks only protect the row being written, not the rows that were _read_ to make the decision.

---

### Solution: Lock All Relevant Rows

Before changing status to inactive, a transaction must **grab locks on all other active doctors' rows** too.

```
Dr. Oz wants to go inactive
  → Acquires locks on ALL active doctor rows (including Dr. Toboggan's)

Dr. Toboggan also wants to go inactive at the same time
  → Tries to acquire the same locks
  → Blocked — Dr. Oz holds them

Dr. Oz completes → releases locks
Dr. Toboggan re-checks → only 1 active doctor → can't go inactive → transaction fails ✓
```

By locking the rows that were _read_ (not just the row being written), the invariant is protected.

## Problem 2: Phantom Writes

**What it is:** A variation of write skew where conflicting transactions insert **brand new rows** — so there are no existing locks to grab.

> **Example:** Two customers try to claim the last cupcake at the same time by each inserting a new row with their name.
> 
> - No cupcake row exists yet → no lock to compete for
> - Both inserts succeed independently
> - Result: **two people own the same cupcake.** Invariant broken.

Standard row-level locking can't help here — you can't lock a row that doesn't exist yet.

---

### Solution: Materializing Conflicts

Pre-populate the database with rows for every possible item _before_ any transaction occurs.

```
Before the bakery opens:
  → Create a row for the cupcake (quantity: 1, owner: null)

Customer A tries to claim it:
  → Grabs lock on the existing cupcake row → writes their name ✓

Customer B tries to claim it at the same time:
  → Tries to grab the same lock → blocked
  → Waits → row already claimed → transaction fails ✓
```

By making the row exist in advance, you create a **physical lock object** that competing transactions can fight over.

---

## Side-by-Side Summary

||Write Skew|Phantom Writes|
|---|---|---|
|**What conflicts**|Writes to separate existing rows|Writes of brand new rows|
|**Why locks fail**|Each transaction locks only its own row|No rows exist to lock|
|**Solution**|Lock _all_ rows involved in the invariant|Materialise conflicts — pre-create the rows|

---

## How This Fits the Isolation Levels

|Isolation Level|Prevents|
|---|---|
|Read Committed|Dirty reads, dirty writes|
|Snapshot Isolation|+ Read skew|
|**Serializable**|**+ Write skew, phantom writes** (most expensive)|

Write skew and phantoms are among the hardest race conditions to handle — full prevention requires serializable isolation.

---

## One-Line Summary

> Write skew breaks invariants when transactions write to separate rows based on stale shared state — fixed by locking all relevant rows. Phantoms are the same problem with new rows instead, fixed by pre-creating rows so locks can exist before the conflict happens.
