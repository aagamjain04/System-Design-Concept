

## The Core Idea

Full serializability (preventing all race conditions) is expensive — concurrent transactions constantly risk stepping on each other.

**Actual Serial Execution** sidesteps the problem entirely:

> Run all transactions **sequentially on a single core**. No concurrency = no race conditions. Isolation is free.

Real-world example: **VoltDB**

---

## The Catch

Single-threaded means **any slow operation blocks everything**. The entire queue stalls behind it.

To make this viable, you have to eliminate the two biggest bottlenecks:

---

## Optimisation 1: Keep Everything in Memory

**Problem:** Disk I/O is far too slow for a single-threaded queue — one disk read stalls every other transaction waiting behind it.

**Solution:** Store all data in RAM. Use fast in-memory indexes:

- **Hash indexes** → O(1) single-key lookups
- **BSTs (Red-Black / AVL trees)** → O(log N) range queries

||Pros|Cons|
|---|---|---|
|**In-Memory Storage**|Much faster than disk|Expensive, limited space, less durable|

> You can add a WAL for durability, but writing to disk reintroduces slowdowns — a deliberate trade-off.

---

## Optimisation 2: Stored Procedures (Reduce Network Overhead)

**Problem:** Sending full SQL queries over the network on every operation is slow. Every byte transferred ties up the single thread.

**Solution:** Pre-load functions (**stored procedures**) onto the database before the app starts. Instead of sending a whole SQL script, the app sends tiny parameters:

```
Without stored procedures:
  → Send full SQL query over network  (large payload, slow)

With stored procedures:
  → Send: productId (32-bit) + quantity (32-bit)  (tiny payload, fast)
  → Database runs the pre-loaded function locally
```

||Pros|Cons|
|---|---|---|
|**Stored Procedures**|Drastically less network data|Hard to version control, sync with app code, and deploy across replicas|

---

## Summary of the Two Optimisations

|Bottleneck|Cause|Fix|
|---|---|---|
|Disk I/O|Reads/writes too slow for single-thread queue|Store everything in RAM|
|Network overhead|Large SQL payloads tie up the single thread|Pre-load stored procedures, send only small parameters|

---

## How This Fits the Isolation Picture

|Approach|Mechanism|Trade-off|
|---|---|---|
|Read Committed|Locks + old value storage|Allows some race conditions|
|Snapshot Isolation|Versioned rows + snapshots|Allows write skew & phantoms|
|**Actual Serial Execution**|**Single-threaded, sequential**|**No race conditions — but throughput limited to one core**|
|Serializable (lock-based)|Locks on all relevant rows|Expensive, complex|

---

## One-Line Summary

> Actual serial execution achieves perfect isolation by removing concurrency entirely — running transactions one at a time on a single core, made viable by keeping data in memory and using stored procedures to minimise the cost of each operation.
