

## The Core Idea

SSI is an **optimistic** approach to achieving full serializability — an alternative to lock-heavy methods like 2PL.

---

## Pessimistic vs. Optimistic Concurrency Control

||Pessimistic (2PL)|Optimistic (SSI)|
|---|---|---|
|**Assumption**|Conflicts are common — lock everything|Conflicts are rare — let transactions run freely|
|**Mechanism**|Grab locks on all rows read and written|No locks — detect conflicts and abort if needed|
|**Cost when no conflict**|Slow — unnecessary locking overhead|Fast — no overhead|
|**Cost when conflict**|Deadlock risk, abort & retry|Abort & retry|
|**Best for**|High-contention workloads|Low-contention workloads|

---

## How SSI Works

Transactions read from a **snapshot** (consistent point-in-time state, same as snapshot isolation). Two additional checks handle conflicts:

### Case 1: Uncommitted Writes Detected at Read Time

```
Transaction A reads a row
  → Notices: another transaction has a pending uncommitted write on this row
  → Proceeds cautiously, flagging the dependency

Before Transaction A commits:
  → Checks: did that uncommitted write eventually commit?
  → Yes → Transaction A's read is now stale → Abort & restart
  → No  → Safe to commit ✓
```

### Case 2: Committed Write After a Dependent Read

```
Transaction A reads row X → row X tracks Transaction A as a dependent

Transaction B comes along → modifies and commits a change to row X

Database checks row X's dependency list:
  → Transaction A was predicated on the old value
  → Transaction A is aborted → retried under new conditions
```

---

## Trade-off: When to Use SSI vs. 2PL

|Scenario|Best Choice|Why|
|---|---|---|
|Users posting to their own profiles|**SSI**|Writes are mostly disjoint — conflicts rare, locking is wasteful|
|Distributed counter (everyone increments the same value)|**2PL**|Constant conflicts — SSI would endlessly abort & retry|

> **Rule of thumb:** Low contention → optimistic (SSI). High contention → pessimistic (2PL).

---

## How This Fits the Full Isolation Picture

|Approach|Model|Key Cost|
|---|---|---|
|Read Committed|Pessimistic|Some race conditions remain|
|Snapshot Isolation|Optimistic reads|Write skew & phantoms remain|
|Actual Serial Execution|Single-threaded|Throughput capped at one core|
|Two-Phase Locking (2PL)|Pessimistic|Deadlocks, resource-heavy|
|**SSI**|**Optimistic**|**Abort overhead under high contention**|

---

## One-Line Summary

> SSI achieves full serializability without locks by letting transactions run freely on snapshots, then aborting any whose reads were invalidated by a concurrent write — fast when conflicts are rare, inefficient when they're not.
