

## The Core Idea

ACID is a set of four guarantees that certain databases provide for how writes are handled. They ensure data stays correct and reliable even when things go wrong (crashes, concurrent users, partial failures).

---

## The Four Properties

### A — Atomicity: "All or Nothing"

Every write in a transaction either **fully succeeds or fully fails**. No partial states.

> **Example:** Sending $10 — money must leave your account _and_ arrive in theirs. If the second step fails, the first is rolled back. Money can't vanish mid-transaction.

---

### C — Consistency: "Always Valid State"

The database never ends up in a corrupted or logically broken state after a failure. It boots back up in a valid, intended state.

> **Example:** A rule says there must always be one security guard on shift. A crash mid-update can't leave the building unguarded — the database ensures invariants are never violated.

---

### I — Isolation: "No Race Conditions"

Concurrent transactions behave as if they ran **one at a time**, even though they're executing simultaneously.

> **Example:** Two users both read a counter at $0 and both try to add $1. Without isolation, both write $1 (not $2). Isolation prevents this by serialising access.

⚠️ Isolation is considered the **hardest and most performance-costly** property to implement correctly.

---

### D — Durability: "Committed = Permanent"

Once a transaction is committed, the data **stays written** — even if the system immediately crashes.  
(Assumes the physical disk itself keeps working.)

---

## Quick Reference

|Letter|Property|One-liner|
|---|---|---|
|**A**|Atomicity|All writes succeed, or none do|
|**C**|Consistency|Database always recovers to a valid state|
|**I**|Isolation|Concurrent transactions don't interfere|
|**D**|Durability|Committed data is permanent|

---

## Implementation: The Write-Ahead Log (WAL)

A, C, and D are all achieved through the WAL (same pattern as indexes):

```
Transaction begins
  → Each action is written sequentially to the WAL on disk
  → All actions complete → "COMMIT" marker written to WAL
  → Changes applied to the database

Crash before COMMIT?
  → Transaction is treated as invalid → not applied → Atomicity preserved

System restarts?
  → Replay all committed entries in the WAL → state rebuilt → Consistency + Durability
```

> **Isolation is the exception** — the WAL doesn't solve it. Isolation requires additional concurrency control mechanisms, which is why it's the hardest and most expensive property.

---

## Why Don't All Databases Use ACID?

Full isolation (also called **serializability**) carries a significant performance penalty — transactions must be carefully coordinated to avoid interfering with each other.

Many databases trade strict ACID guarantees for better throughput, offering weaker isolation levels instead. This is a deliberate systems design choice, not an oversight.

---

## One-Line Summary

> ACID guarantees that database transactions are all-or-nothing, leave the database valid, don't interfere with each other, and persist permanently — with the WAL handling A, C, and D cheaply, and isolation being the hard, expensive one.
