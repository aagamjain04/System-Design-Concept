

## The Context

Databases are multi-threaded — many reader and writer threads run concurrently. Because the exact order of execution is unpredictable, **race conditions** can corrupt results.

> A write is only **committed** once all writes in a transaction succeed. Until then, changes are uncommitted and potentially unsafe to expose.

---

## The Two Race Conditions It Solves

### 1. Dirty Writes

**What it is:** A thread overwrites another thread's _uncommitted_ data.

> **Example:** Two threads both update the same two rows (e.g. purchaser name + delivery address). Their operations interleave — Thread A wins on row 1, Thread B wins on row 2. The result is a broken, mismatched record.

**Solution: Row-Level Locking**

- A thread must acquire an exclusive lock on a row before modifying it
- Other threads block until the lock is released
- Prevents interleaving entirely

**Handling Deadlocks** (where two threads wait on each other forever):

- Require threads to always acquire locks in a **consistent order**, or
- Use **deadlock detection** to force one transaction to surrender its locks

---

### 2. Dirty Reads

**What it is:** A thread reads another thread's _uncommitted_ data.

> **Example:** Transaction A deducts $10 from a balance. Transaction B reads this new lower balance. Transaction A then fails — the deduction is rolled back. Transaction B has now acted on data that never officially existed.

**Two possible solutions:**

|Approach|How it works|Problem|
|---|---|---|
|**Row-Level Locking**|Block reads until write commits|Slow writes block all reads → massive bottleneck|
|**Store Old Values**✅|Keep the old value visible until commit, then flip a pointer|Slightly more storage, but fast — **preferred**|

The store-old-values approach means readers always see the last committed state, never an in-progress write.

```
Write begins on a row
  → Old value kept in storage
  → Readers continue seeing old value
  → Write commits → pointer flips to new value
  → Readers now see new value
```

---

## What is Read Committed Isolation?

A database achieves **Read Committed Isolation** when it protects against **both**:

|Protection|Mechanism|
|---|---|
|No dirty writes|Row-level locking|
|No dirty reads|Store old values until commit|

---

## Important Caveat

Read Committed Isolation only solves these two specific race conditions. There are more complex race conditions (e.g. non-repeatable reads, phantom reads) that require stronger isolation levels beyond read committed.

---

## One-Line Summary

> Read Committed Isolation prevents threads from reading or writing uncommitted data — using row-level locks to block dirty writes, and storing old values to serve clean reads without expensive locking.
