 

## The Core Idea

All **writes** go to one **leader** node. Multiple **followers** receive updates asynchronously via a replication log. **Reads** can come from any node.

```
Client writes → Leader only
Client reads  → Leader or any follower

Leader → replication log → Follower A
                         → Follower B
                         → Follower C
```

Replication is **asynchronous** in ~99% of real cases. Synchronous is only used when stale reads are strictly unacceptable (rare).

---

## Key Advantages

|Advantage|How|
|---|---|
|**Durability**|Data survives if one node or data centre goes down|
|**Read throughput**|Each follower added linearly scales read capacity|
|**Lower latency**|Place followers geographically near users|
|**Simplicity**|Easy to set up — just attach followers to an existing database|

---

## Failure Scenarios

### 1. Follower Failure — Easy ✅

Each follower tracks its last processed position in the replication log.

```
Follower crashes at log position 50
  → Comes back online
  → Requests positions 51 onwards from leader
  → Catches up and resumes normally
```

---

### 2. Leader Failure — Hard ❌

Three serious problems arise:

**Problem 1: False Positives (Is the leader actually dead?)** Networks are unreliable — a follower might assume the leader is dead due to a bad connection when it's actually fine. There's no easy way to definitively confirm a leader is down.

**Problem 2: Lost Writes**

```
Leader accepts writes up to log position 80
Followers only received up to position 70
Leader dies → follower promoted to new leader
  → Writes 71–80 are permanently lost
```

Asynchronous replication means accepted writes can vanish if the leader dies before they propagate.

**Problem 3: Split Brain**

```
Leader dies → Follower A promoted to new leader
Original leader recovers → now two leaders exist
Both send conflicting writes to followers
  → Corrupted, inconsistent data
```

---

## Architectural Considerations

### Distributed Consensus

To handle leader failure safely, the system needs all nodes to **agree** on:

- Who the current leader is
- Whether a node is definitively up or down

Without distributed consensus, split brain and lost writes are unavoidable.

### Write Bottleneck — The Core Limitation

All writes must pass through one node. Single leader replication **cannot scale write throughput** — adding more followers doesn't help.

Not suitable for write-heavy applications. Those require a different replication strategy (multi-leader or leaderless).

---

## Summary

||Single Leader Replication|
|---|---|
|Write scalability|❌ Bottlenecked at one node|
|Read scalability|✅ Linear with each follower added|
|Follower failure|✅ Easy — replay from log position|
|Leader failure|❌ Hard — lost writes, split brain, false positives|
|Setup complexity|✅ Simple|
|Best for|Read-heavy workloads with moderate write volume|

---

## One-Line Summary

> Single leader replication routes all writes through one node and scales reads with followers — simple and effective for read-heavy systems, but fundamentally limited by write throughput and complex, dangerous leader failure scenarios.
