

## The Core Idea

An **LSM Tree (Log-Structured Merge-Tree)** combines:

- An **in-memory** balanced BST (Red-Black, AVL, or B-Tree) for fast writes
- **On-disk SSTables** for persistent, sorted storage

Developed as a middle ground between hash indexes and B-Trees — for datasets too large for RAM but needing faster writes than a B-Tree can offer.

---

## How It Works — Step by Step

### Writes

All incoming writes go to the **in-memory BST** → fast O(log N)

### Flushing to Disk

When the in-memory tree hits a size threshold:

```
In-memory BST fills up
  → In-order traversal (keys already sorted)
  → Written to disk as an SSTable (Sorted String Table)
  → Memory is cleared, ready for new writes
```

### SSTables

- Keys stored in **sorted order** on disk
- **Immutable** — once written, never changed

### Updates & Deletes

Since SSTables can't be edited:

- **Update** → write the new value to the in-memory tree (newer entries always win)
- **Delete** → write a **tombstone marker** for that key (signals "this is gone")

Both are handled lazily — the old data gets cleaned up during compaction.

### Reads

```
Search in-memory BST first
  → Found? Return it.
  → Not found? Search SSTables on disk, newest first
      → Use binary search within each file (keys are sorted) → O(log N)
```

---

## The Four Key Optimizations & Trade-offs

### 1. Read Optimizations (Reads are the weak spot)

Searching multiple SSTable files on disk is slow. Two fixes:

|Tool|What it does|
|---|---|
|**Sparse Index**|Stores disk locations of a subset of keys → narrows down where to binary search|
|**Bloom Filter**|Probabilistic filter — instantly tells you if a key is _definitely not_ in a file → skip it entirely|

---

### 2. Compaction

Over time, the same key might appear across dozens of SSTables with old values and tombstones → wasted space.

> **Compaction** — a background process that merges multiple SSTables into one, keeping only the most recent value per key and dropping deleted entries.

Merge is efficient: since all SSTables are already sorted, merging is O(N) (like merging sorted lists).

---

### 3. Fast Writes, Slower Reads

|Operation|LSM Tree|B-Tree|
|---|---|---|
|**Writes**|✅ Very fast (RAM first)|Slower (disk splits + WAL)|
|**Range queries**|Supported, but slower (multiple files)|✅ Fast (one sorted structure)|
|**Single reads**|Slower (check memory + multiple files)|✅ Faster (one traversal)|

---

### 4. Durability (WAL — same pattern)

In-memory BST is wiped on power loss.

> **Write Ahead Log (WAL)** — every write is logged sequentially to disk _before_ touching the in-memory tree.  
> On crash → replay the WAL → in-memory tree is rebuilt.

## All Three Indexes — Full Comparison

||Hash Index|B-Tree|LSM Tree|
|---|---|---|---|
|Storage|RAM only|Disk|RAM + Disk|
|Dataset size|Limited by RAM|Unlimited|Unlimited|
|Single lookup|**O(1)**|O(log N)|O(log N) slower|
|Range query|❌|✅ O(log N)|✅ (slower)|
|Write speed|Fast|Slower|**Fastest**|
|Durability|WAL|WAL|WAL|
|Extra complexity|—|—|Compaction, bloom filters|

---

## When to Use an LSM Tree ✅

Use it when **both** are true:

1. Your dataset is **too large to fit in RAM**
2. Your application is **write-heavy** and can tolerate slightly slower reads

---

## One-Line Summary

> An LSM Tree writes everything to a fast in-memory BST, flushes it to immutable sorted SSTables on disk, and uses compaction + bloom filters to stay efficient — trading read simplicity for the fastest possible write speeds.
