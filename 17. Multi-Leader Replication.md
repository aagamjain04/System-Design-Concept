

## The Core Idea

Multiple **leader** nodes each accept writes. Leaders sync with each other via replication logs. **Followers** receive propagated writes asynchronously and are read-only.

```
Client A → Leader 1 (Europe)   ↘
                                 → sync with each other → Followers
Client B → Leader 2 (US)      ↗
```

Solves the single write bottleneck of single-leader replication.

---

## Key Advantages

|Advantage|How|
|---|---|
|**Write throughput**|Writes distributed across multiple nodes — no single bottleneck|
|**Lower write latency**|Users write to their nearest regional leader — less network transit|

---

## Replication Topologies

How leaders propagate writes to each other:

### Circle Topology

Each leader passes writes to the next in a chain.

- ❌ **Single point of failure** — one node down breaks the entire chain

### Star Topology

A central hub node distributes writes to all others.

- ✅ Outer node fails → system survives
- ❌ **Hub fails → all inter-leader communication severed**

### All-to-All Topology

Every leader talks directly to every other leader.

- ✅ Most resilient to node failures
- ❌ **Causality issues** — writes can arrive out of order (write B arrives before write A that it depends on)

### Preventing Infinite Loops

In all-to-all setups, writes risk bouncing around the network indefinitely.

> **Fix:** Tag each write with the set of nodes that have already seen it. A node that recognises a write it's already processed stops propagating it.

---

## The Biggest Challenge: Write Conflicts

Two users write to the same key on different leaders simultaneously → conflict.

### Conflict Resolution Strategies

#### 1. Conflict Avoidance

Force all writes for a specific key to always go to the same leader.

- ❌ Effectively partitions writes — eliminates the throughput and latency advantages of multi-leader

---

#### 2. Last Write Wins (LWW)

Attach a timestamp to every write — the latest timestamp wins.

**Why sender timestamps fail:**

- Users can manually set their device clock to the future → always win every conflict

**Why receiver timestamps fail:**

|Problem|Cause|
|---|---|
|**Clock skew**|Quartz crystals in hardware drift over time due to temperature and wear → local time becomes inaccurate|
|**NTP corrections**|Servers use Network Time Protocol to sync with GPS clocks — but network delays make adjustments imprecise, sometimes causing a server clock to jump _backwards_|

```
Server A clock: 12:00:00.500
Server B clock: 12:00:00.498  (slightly behind due to drift)

User on Server B writes at "12:00:00.498"
User on Server A writes at "12:00:00.500"

LWW: Server A's write wins
  → But what if Server A's clock was actually fast?
  → A perfectly valid newer write is silently discarded
```

**Conclusion:** Clocks can never be perfectly synchronised. LWW inevitably causes valid writes to be silently lost due to microscopic clock skew. There is no reliable way to perfectly order concurrent writes across distributed nodes.

## Single Leader vs. Multi-Leader

||Single Leader|Multi-Leader|
|---|---|---|
|Write throughput|❌ Bottlenecked|✅ Scales with leaders|
|Write latency (global)|❌ All writes to one region|✅ Write to nearest leader|
|Conflict resolution|✅ No conflicts (one writer)|❌ Hard problem|
|Setup complexity|✅ Simple|❌ Complex|
|Best for|Read-heavy, moderate writes|Write-heavy, global user base|

---

## One-Line Summary

> Multi-leader replication removes the write bottleneck by letting multiple nodes accept writes, but introduces the hard problem of concurrent write conflicts — which conflict avoidance partially addresses and LWW cannot reliably solve due to unavoidable clock skew.
