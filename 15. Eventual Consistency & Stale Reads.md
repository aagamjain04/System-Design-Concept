

## The Core Trade-off

**Strong consistency** — every replica is always in sync before confirming writes.  
→ Too slow and expensive at global scale.

**Eventual consistency** — write to one or two nodes, replicas catch up in the background.  
→ Fast, but clients may read from a stale replica.

You can't eliminate all stale reads — but you can prevent the ones that frustrate users.

---

## The Three Problems & Solutions

### 1. Reading Your Own Writes

**The issue:** User updates their profile → page reloads from a stale replica → user sees their old data and thinks the update failed.

**Solutions:**

|Approach|How it works|
|---|---|
|**Time-windowed reads**|For N seconds after a write, force reads to come from the node that processed the write|
|**Timestamp verification**|Attach a timestamp to the write (e.g. T=100). Before reading from any replica, check: "Have you seen up to T=100?" If not, skip to one that has|

---

### 2. Monotonic Reads (Time Going Backwards)

**The issue:** User hits refresh multiple times, hitting different replicas at different lag levels.

```
Refresh 1 → fast replica  → sees Message 3
Refresh 2 → slow replica  → sees Message 2   ← goes backwards
Refresh 3 → slower replica → sees Message 1  ← goes backwards again
```

Time appears to run in reverse. Deeply confusing UX.

**Solution: Pin the user to one replica**

- Route each user to the same replica every time
- Even if that replica is lagging, it only moves forward — reads are monotonic
- **Implementation:** `assigned replica = User ID % Total Number of Replicas`

---

### 3. Consistent Prefix Reads (Broken Causal Order)

**The issue:** Causally linked messages end up on different partitions. The reply arrives before the original question.

```
Partition A: "want to grab lunch?"   ← replicates slowly
Partition B: "sure!"                 ← replicates fast

Third user sees:
  "sure!"              ← no context
  [question arrives later]
```

**Solution: Keep causally dependent writes on the same partition**

- If a message is a reply, route both the original and the reply to the same partition
- Same partition = same replication order = causal order preserved

---

## Why These Three Matter Most

|Problem|Why users notice|Why they don't notice other stale reads|
|---|---|---|
|Reading own writes|They know what they just did|They don't know a post exists yet|
|Monotonic reads|Time going backward is disorienting|A delayed feed post isn't missed|
|Consistent prefix reads|Missing context breaks meaning|Isolated updates make sense alone|

The key insight: users are tolerant of _missing_ new data, but not of _broken_ or _contradictory_ data they've already seen.

---

## One-Line Summary

> Eventual consistency trades stale reads for speed — the three jarring failure cases (your own write disappearing, time going backwards, and replies without context) are fixed by routing reads to the right node, pinning users to one replica, and keeping causally linked writes on the same partition.
