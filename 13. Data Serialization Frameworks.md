

## The Problem

When storing or transmitting objects, you need an efficient encoding format. Two common approaches fall short:

|Approach|Problem|
|---|---|
|**SQL / relational tables**|Fetching one full record requires expensive joins|
|**JSON / XML (plain text)**|Human-readable, but field names are repeated for every row → massive overhead, not type-safe|

```json
{"name": "Alice", "age": 30, "email": "alice@example.com"}
{"name": "Bob",   "age": 25, "email": "bob@example.com"}
```

Writing `"name"`, `"age"`, `"email"` millions of times wastes disk and bandwidth.

---

## Protocol Buffers (Protobuf) & Apache Thrift

Both use **pre-defined schemas** with numbered field tags to eliminate string overhead.

### How It Works

```
Schema definition:
  field 1 = name   (string)
  field 2 = age    (int)
  field 3 = email  (string)

Encoded row:
  1: "Alice"  2: 30  3: "alice@example.com"
  ↓
  Stored in binary — tag numbers take far less space than repeated key strings
```

### Pros & Cons

||Protobuf / Thrift|
|---|---|
|✅ Type-safe|Yes|
|✅ Space efficient|Binary — no repeated field names|
|✅ Self-documenting|Schema files serve as clear API docs|
|❌ Human-readable|No — binary only|
|❌ CPU cost|Serialize + deserialize penalty|
|❌ Schema required upfront|Must know structure at compile time|

---

### Schema Evolution

Data changes over time. Rules for adding fields safely:

- **New fields must be optional (nullable)** — never required without a default
- If you add a required field, old rows without it become invalid and break the app
- Optional fields ensure **backwards and forwards compatibility**:
    - Old code reads new row → ignores unknown field → works fine
    - New code reads old row → missing field → uses default value → works fine

---

## Apache Avro

**Problem Avro solves:** You don't always know the schema in advance — e.g. ingesting data from external companies into a data lake (Hadoop).

- No numeric tags — uses **column names** instead
- Generates schemas **dynamically** from incoming data
- Ideal for data lakes / warehouses where schemas vary by source

---

### Reader vs. Writer Schema Reconciliation

Server A (writer) and Server B (reader) may expect different formats. Avro handles the mismatch:

|Situation|Avro's behaviour|
|---|---|
|Field exists in both schemas|Mapped and read normally|
|Writer has a field the reader doesn't know|Reader ignores it|
|Reader expects a field the writer didn't include|Reader fills it with a default value|

```
Writer schema:  name, age, net_worth
Reader schema:  name, age, attractiveness

Result:
  name      → matched ✓
  age       → matched ✓
  net_worth → unknown to reader → ignored
  attractiveness → missing from writer → filled with default (e.g. 0)
```

> Best practice: store Avro schemas in a **centralised schema registry** so all services stay in sync.

## Comparison: All Three Approaches

||JSON / XML|Protobuf / Thrift|Avro|
|---|---|---|---|
|Human-readable|✅|❌|❌|
|Type-safe|❌|✅|✅|
|Space efficient|❌|✅|✅|
|Schema required upfront|No|✅ Yes|No — dynamic|
|Schema mismatch handling|Manual|Manual|✅ Automatic|
|Best for|APIs, debugging|Services with known schemas|Data lakes, unknown/variable schemas|

---

## One-Line Summary

> Serialization frameworks like Protobuf and Thrift replace verbose JSON field names with compact binary tags for efficiency, while Avro goes further by generating schemas dynamically and automatically reconciling mismatches between writer and reader — ideal when data sources are unpredictable.
