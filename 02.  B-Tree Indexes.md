
## The Core Idea

B-Trees live **entirely on disk** (unlike hash indexes which live in RAM).  
They are a **self-balancing tree** that maps ranges of keys to pointers, which guide the database to the exact row on disk.

---

## How B-Trees Work

### Structure

- The tree is made of **nodes (pages)** — each node is large (~256 MB)
- Large nodes = tree stays **short/shallow** = fewer physical disk jumps = faster reads

### Reading & Updating

Traverse down the tree following key ranges → reach the target node → read or update **in place**.

### Writing (Insertion)

```
New key comes in
  → Find the correct leaf node (keys must stay in sorted order)
  → Node has space?  → Write it. Done.
  → Node is full?    → Split into two nodes
                       → Update parent's references
                       → Parent full? → Split parent too
                       → Repeat up to root if needed  (cascading split)
```

> The tree always stays **perfectly balanced** after every write.

---

## The Four Key Properties

### 1. Range Queries ✅

Keys are always stored in **sorted order** within nodes.  
To answer _"give me all names between E and L"_:

- Traverse to the start of the range
- Read adjacent keys sequentially — no random jumps needed

This is B-Trees' biggest advantage over hash indexes.

---

### 2. No RAM Constraint ✅

Stored entirely on disk → keys do **not** need to fit in memory.  
Handles far larger datasets than a hash index can.

---

### 3. Durability (WAL again)

A cascading split interrupted by a power cut can leave references broken → **corrupted tree**.

The fix is the same as hash indexes:

> **Write Ahead Log (WAL)** — every action is logged sequentially to disk _before_ the B-Tree is modified.  
> On crash → replay the WAL → tree recovers to a consistent state.

---

### 4. Slower Than Hash Indexes ⚠️

- **Reads**: Reasonably fast (shallow tree = few disk jumps), but disk is always slower than RAM
- **Writes**: Can be expensive — one insert may trigger:
    1. Write to WAL
    2. Multiple node splits
    3. Traversal back up the tree to update references

---

## Quick Trade-off Comparison

||B-Tree Index|Hash Index|
|---|---|---|
|Storage|Disk ✅|RAM only|
|Dataset size|Unlimited ✅|Limited by RAM|
|Single lookup|O(log N)|**O(1)** ✅|
|Range query|**O(log N)** ✅|❌ Useless|
|Write cost|Higher (splits + WAL)|Lower|
|Durability|WAL|WAL|

---

## When to Use a B-Tree Index ✅

Use it when **either** condition is true:

1. Your dataset is **too large to fit in RAM**
2. Your application **needs range queries**

---

## One-Line Summary

> A B-Tree index trades the raw O(1) speed of a hash index for disk-based storage, range query support, and no memory limits — with a WAL keeping it safe from crashes.
